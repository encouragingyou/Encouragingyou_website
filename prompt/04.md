# Prompt 04 - State Management + Interaction Orchestration Contract

## Shared Execution Context

These prompts assume the repo root contains a `/source` directory.
Run them strictly in order.
Each prompt should be executed only after the previous prompt has completed and committed its outputs.

## Global Execution Rules

- Treat `/source/blurpint` as the current prototype, not the final architecture.
- Treat `/source/encouragingyou-site-look-and-feel.md`, `/source/EncouragingYou Website Information .pdf`, and `/source/media_attachment/**` as primary inputs.
- Reuse source assets by reference; do not duplicate the raw brief into code comments or docs unless normalization genuinely requires it.
- Make reasoned technical decisions autonomously. Record assumptions when facts are missing.
- Prefer the lightest robust implementation that supports performance, accessibility, maintainability, privacy, and future editorial growth.
- Preserve working behavior until replacement behavior is implemented and verified.
- At the end of each prompt, leave a short handoff note for the next prompt inside the repo.

Design and implement the behavioral architecture for the website. This is a content-first site, so the objective is not to add heavy client state for its own sake. The objective is to identify every interactive or derived state, assign clear ownership, and implement the lightest robust state model that can scale.

## Inputs

- Everything from Prompts 01-03
- The current interaction logic in `/source/blurpint/assets/js/site.js`
- Any forms, disclosure patterns, schedule logic, or interactive modules already present in the migrated build

## Expected Processing States

1. **State inventory**
   - Enumerate all user-visible, derived, persistent, URL-driven, and transient states across the site.
   - At minimum consider: navigation state, mobile menu state, active route state, form entry/validation/submission state, FAQ/disclosure state, session recurrence calculation state, event structured-data derivation state, filter/search state for future events or updates, consent state, reduced-motion state, and loading/error/empty states.

2. **Ownership mapping**
   - For each state, determine whether it belongs in local component state, URL state, shared application state, server-derived state, or persisted client state.
   - Explicitly reject unnecessary global state where local state is sufficient.

3. **Transition modeling**
   - Define the state transitions, events, side effects, and failure states for each interactive subsystem.
   - Express critical flows as deterministic state machines or transition tables where complexity justifies it.

4. **Pure-domain extraction**
   - Pull recurrence logic, date derivation, and other domain rules out of opaque DOM manipulation into pure, testable utilities.
   - Ensure structured data generation and scheduling behavior are driven from canonical content and tested logic.

5. **Implementation**
   - Build the minimal state orchestration layer required by the chosen stack.
   - Add validation, submission, and success/error handling patterns for current and upcoming forms.
   - Add resilient defaults for no-JS, reduced-motion, and partial-failure scenarios where relevant.

6. **Verification**
   - Add unit or integration tests for stateful logic and edge cases.
   - Confirm that interactions remain accessible, predictable, and maintainable.

## Strategic Outputs

Create all of the following:

- `docs/state/04-state-architecture.md`
- `docs/state/04-transition-maps.md`
- Pure utility modules for recurrence/date/derived data logic
- The chosen in-app state implementation for interactive UI concerns
- Validation and submission state handling for current forms or form abstractions
- Automated tests for non-trivial behavioral logic
- `docs/handovers/04-to-05.md`

## Non-Negotiable Constraints

- Do not use a global store unless the state inventory proves it is necessary.
- Do not leave critical interaction rules hidden in scattered DOM selectors and side effects.
- All derived state should trace back to canonical inputs.
- The behavior model must remain compatible with accessibility, graceful degradation, and future feature growth.

## Done When

- An engineer can point to a clear owner and transition model for every meaningful interactive behavior.
- Derived logic is testable and no longer dependent on brittle page-specific scripts.
