# Prompt 05 - Playwright UI/UX Validation Layer for User Flows, Structural Integrity, and Responsive Behavior

## Shared Execution Context

These prompts assume the repo root contains a `/source` directory.
Run them strictly in order.
Each prompt should be executed only after the previous prompt has completed and committed its outputs.

## Global Execution Rules

- Treat `/source/blurpint` as the current prototype, not the final architecture.
- Treat `/source/encouragingyou-site-look-and-feel.md`, `/source/EncouragingYou Website Information .pdf`, and `/source/media_attachment/**` as primary inputs.
- Reuse source assets by reference; do not duplicate the raw brief into code comments or docs unless normalization genuinely requires it.
- Make reasoned technical decisions autonomously. Record assumptions when facts are missing.
- Prefer the lightest robust implementation that supports performance, accessibility, maintainability, privacy, and future editorial growth.
- Preserve working behavior until replacement behavior is implemented and verified.
- At the end of each prompt, leave a short handoff note for the next prompt inside the repo.

Architect an automated validation layer using Playwright. This is not just a smoke test pass. It is the beginning of a durable UI/UX quality system that protects the site as later prompts add more routes, content, motion, forms, and legal/compliance features.

## Inputs

- Everything from Prompts 01-04
- The current and migrated route structure
- Existing Playwright artifacts in `/source/blurpint/.playwright-cli/**`
- The design and behavior expectations defined in prior prompt outputs

## Expected Processing States

1. **Critical-flow modeling**
   - Identify the highest-value launch journeys and convert them into testable end-to-end flows.
   - At minimum cover flows such as:
     - Home -> Sessions -> CV support detail -> calendar or enquiry path
     - Home -> Sessions -> Youth club detail -> calendar or enquiry path
     - Home -> Get support/contact path -> privacy visibility
     - Home -> Get involved -> volunteer/partner/referral path
     - Home -> Safeguarding -> contact/escalation route

2. **Structural contract definition**
   - Define site-wide structural assertions: page title presence, single H1 expectation, landmark integrity, nav/footer integrity, internal-link validity, route availability, no major console errors, and no catastrophic layout breakage.
   - Add assertions for critical legal and trust surfaces when they are present.

3. **Responsive matrix design**
   - Define a viewport matrix covering at least mobile, tablet, laptop, and large desktop.
   - For each breakpoint, verify readable layout, no destructive overlap, no horizontal overflow, reachable navigation, functional CTAs, and sensible content order.

4. **Behavioral and accessibility smoke layer**
   - Add checks for keyboard navigation on critical controls, reduced-motion behavior where relevant, form validation visibility, and high-level accessibility regressions.
   - Catch obvious broken states such as hidden focus, inaccessible menus, or broken submission paths.

5. **Implementation architecture**
   - Create the Playwright configuration, test organization, fixtures, helpers, and reusable assertions.
   - Prefer clear, maintainable test architecture over clever abstraction.
   - Capture artifacts on failure and generate human-readable reports.

6. **Quality-gate definition**
   - Define which tests are blocking and which are advisory.
   - Add scripts so future prompts can run the suite locally and in automation without ambiguity.

## Strategic Outputs

Create all of the following:

- `docs/qa/05-playwright-validation-architecture.md`
- `playwright.config.*`
- `tests/e2e/**` or the equivalent test directory in the chosen stack
- Shared Playwright fixtures/helpers/assertion utilities
- Baseline tests for critical launch flows
- Structural integrity tests for core routes
- Responsive behavior tests across the viewport matrix
- Reporter and artifact configuration for screenshots, traces, and videos where useful
- Package scripts or task-runner commands for local execution
- `docs/handovers/05-to-06.md`

## Non-Negotiable Constraints

- Do not write tests that only confirm implementation details invisible to users.
- Do not limit the suite to happy-path clicks; include structural and responsive failure detection.
- Keep tests stable, deterministic, and scoped to real launch risk.
- Make sure the validation layer can expand as later prompts add events, legal pages, consent flows, richer forms, and animation polish.

## Done When

- The repo has a real automated UI/UX validation foundation.
- A future agent can extend the suite confidently as more features are built.
- Regressions in user flow, layout structure, and responsive behavior become visible early instead of surfacing at launch.
