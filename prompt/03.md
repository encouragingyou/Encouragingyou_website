# Prompt 03 - Component Abstraction + Design System Extraction

## Shared Execution Context

These prompts assume the repo root contains a `/source` directory.
Run them strictly in order.
Each prompt should be executed only after the previous prompt has completed and committed its outputs.

## Global Execution Rules

- Treat `/source/blurpint` as the current prototype, not the final architecture.
- Treat `/source/encouragingyou-site-look-and-feel.md`, `/source/EncouragingYou Website Information .pdf`, and `/source/media_attachment/**` as primary inputs.
- Reuse source assets by reference; do not duplicate the raw brief into code comments or docs unless normalization genuinely requires it.
- Make reasoned technical decisions autonomously. Record assumptions when facts are missing.
- Prefer the lightest robust implementation that supports performance, accessibility, maintainability, privacy, and future editorial growth.
- Preserve working behavior until replacement behavior is implemented and verified.
- At the end of each prompt, leave a short handoff note for the next prompt inside the repo.

Using the normalized architecture and data layer, convert the visual and structural patterns into a reusable component system fit for a premium production website.

## Inputs

- Everything from Prompts 01 and 02
- The existing visual implementation in `/source/blurpint/**`
- The design principles and look-and-feel rules in `/source/encouragingyou-site-look-and-feel.md`
- The illustration and icon assets in `/source/media_attachment/**` and `/source/blurpint/images/**`, `/source/blurpint/icons/**`

## Expected Processing States

1. **Pattern mining**
   - Inspect the current pages and identify repeated layout, content, and interaction patterns.
   - Separate one-off page copy from reusable UI structures.

2. **Token extraction**
   - Derive colour, spacing, radius, typography, shadow, motion, and breakpoint tokens from the blueprint and the strongest parts of the existing build.
   - Normalize them into a token system that can be reused consistently.

3. **Component taxonomy design**
   - Define primitives, composites, sections, and page shells.
   - At minimum consider: header, navigation, footer, buttons, links, cards, media blocks, trust strips, FAQ blocks, forms, fieldsets, disclosure notes, legal notice blocks, session summaries, event cards, hero variants, CTA panels, breadcrumbs, and empty/error states.

4. **Abstraction boundary selection**
   - Decide which elements should become reusable components, which should remain template-level compositions, and which should stay content-driven.
   - Avoid both duplication and premature over-abstraction.

5. **Implementation**
   - Build the component system inside the chosen application architecture.
   - Refactor repeated prototype markup into reusable components and shared layouts.
   - Introduce responsive media wrappers, icon handling, and accessible image/disclosure treatments.

6. **Proof composition**
   - Rebuild representative page sections using the new component system to prove that the abstraction works in practice.
   - Use real normalized content from Prompt 02 rather than re-hardcoding page text.

## Strategic Outputs

Create all of the following:

- `docs/design-system/03-token-system.md`
- `docs/design-system/03-component-inventory.md`
- `docs/design-system/03-template-composition-map.md`
- Real token files in code
- Real reusable layout and UI components in code
- At least one representative route or route section refactored to use the new component system end to end
- `docs/handovers/03-to-04.md`

## Non-Negotiable Constraints

- Do not keep duplicating page shells, navs, footers, forms, or CTA sections page by page.
- Do not treat the current CSS as sacred; preserve the visual quality while improving system coherence.
- Every abstraction must help maintainability, consistency, accessibility, or content reuse.
- Build for responsive behavior from the start rather than retrofitting it later.

## Done When

- The codebase has a real component architecture rather than a collection of isolated pages.
- A future agent can assemble new launch pages from reusable building blocks with minimal duplication.
