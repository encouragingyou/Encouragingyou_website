# Prompt 02 - Information Architecture + Structured Content/Data Integration Engine

## Shared Execution Context

These prompts assume the repo root contains a `/source` directory.
Run them strictly in order.
Each prompt should be executed only after the previous prompt has completed and committed its outputs.

## Global Execution Rules

- Treat `/source/blurpint` as the current prototype, not the final architecture.
- Treat `/source/encouragingyou-site-look-and-feel.md`, `/source/EncouragingYou Website Information .pdf`, and `/source/media_attachment/**` as primary inputs.
- Reuse source assets by reference; do not duplicate the raw brief into code comments or docs unless normalization genuinely requires it.
- Make reasoned technical decisions autonomously. Record assumptions when facts are missing.
- Prefer the lightest robust implementation that supports performance, accessibility, maintainability, privacy, and future editorial growth.
- Preserve working behavior until replacement behavior is implemented and verified.
- At the end of each prompt, leave a short handoff note for the next prompt inside the repo.

Use the architecture baseline from Prompt 01 to transform scattered content into a normalized, implementation-ready content system.

## Expected Processing States

1. **Audience and page-intent extraction**
   - Infer the launch audiences, their user journeys, and the page intents the site must satisfy.
   - Convert this into a launch sitemap, navigation model, footer model, and page-template inventory.

2. **Domain decomposition**
   - Identify the core content entities needed for a premium, scalable build.
   - At minimum consider: site settings, navigation, contact methods, programmes, sessions, recurring schedules, events, updates/news, FAQs, involvement routes, safeguarding branches, legal pages, reusable CTAs, trust signals, media assets, and notices/disclosures.

3. **Source-of-truth resolution**
   - Determine where each entity should live and how it should be edited.
   - Eliminate accidental duplication between hardcoded HTML, JS objects, calendar files, and future page content.
   - Promote one canonical source for each domain concept.

4. **Schema design**
   - Define the structured schema for each content type.
   - Model required fields, optional fields, editorial defaults, validation rules, relationships, and derived fields.
   - Include missing-but-essential placeholders where the business has not yet provided final facts.

5. **Ingestion and normalization**
   - Create a repeatable content integration layer that imports or rewrites the current prototype content into the new schema.
   - Convert existing session metadata and calendar logic into normalized domain data rather than leaving it embedded in ad hoc page scripts.
   - Map illustrations and icons to reusable media references with alt-text strategy.

6. **Launch vs later partitioning**
   - Split content entities into launch-ready, placeholder, and phase-two buckets.
   - Ensure the data model supports later expansion without breaking the launch site.

## Non-Negotiable Constraints

- Do not leave key business entities trapped inside duplicated HTML fragments when they should be structured data.
- Do not invent fake business facts; use explicit placeholders and assumption logs.
- Keep the content layer flexible enough for future CMS adoption even if launch uses local content files.
- Derived data such as recurring session dates must be generated from structured inputs, not duplicated manually across pages.

## Done When

- The site has a coherent content model that can drive routes, templates, structured data, forms, and future editorial workflows.
- A future agent can build pages from the normalized data layer instead of scraping the old prototype repeatedly.
