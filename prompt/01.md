# Prompt 01 - Foundation Audit -> Gap Matrix -> Target Architecture Baseline

## Shared Execution Context

These prompts assume the repo root contains a `/source` directory.
Run them strictly in order.
Each prompt should be executed only after the previous prompt has completed and committed its outputs.

## Global Execution Rules

- Treat `/source/blurpint` as the current prototype, not the final architecture.
- Treat `/source/encouragingyou-site-look-and-feel.md`, `/source/EncouragingYou Website Information .pdf`, and `/source/media_attachment/**` as primary inputs.
- Reuse source assets by reference; do not duplicate the raw brief into code comments or docs unless normalization genuinely requires it.
- Make reasoned technical decisions autonomously. Record assumptions when facts are missing.
- Prefer the lightest robust implementation that supports performance, accessibility, maintainability, privacy, and future editorial growth.
- Preserve working behavior until replacement behavior is implemented and verified.
- At the end of each prompt, leave a short handoff note for the next prompt inside the repo.

You are taking over a partially complete website prototype. Your job is not to cosmetically tweak the current build. Your job is to infer the best production-grade architecture for the final site using the real repository state.

## Inputs

- `/source/blurpint/**`
- `/source/encouragingyou-site-look-and-feel.md`
- `/source/EncouragingYou Website Information .pdf`
- `/source/media_attachment/**`
- Any existing screenshots or Playwright artifacts inside `/source/blurpint/.playwright-cli/**`

## Expected Processing States

1. **Repository census**
   - Enumerate all current routes, HTML pages, CSS, JS, images, icons, fonts, calendar files, metadata files, legal pages, forms, and automation artifacts.
   - Identify what is hardcoded, what is derived, and what is reusable.

2. **Intent extraction**
   - Read the blueprint and PDF brief.
   - Distill the intended audiences, trust requirements, page goals, conversion flows, content categories, and missing launch-critical pages or systems.

3. **Delta analysis**
   - Compare the implemented prototype to the intended product.
   - Create a gap matrix covering: information architecture, content coverage, data modeling, legal/compliance, UX polish, accessibility, responsiveness, forms, automation, performance, SEO, and deployment readiness.

4. **Architecture optioning**
   - Evaluate multiple implementation directions rather than assuming the current static approach is final.
   - Score candidate architectures against the actual needs of this project: content-heavy pages, high trust requirements, structured session/event data, legal pages, maintainable components, future CMS compatibility, privacy-conscious defaults, and strong performance.

5. **Decision synthesis**
   - Select the target architecture.
   - Define rendering strategy, routing strategy, content source strategy, styling strategy, testing strategy, and migration boundaries.
   - Explicitly justify why the chosen architecture is stronger than continuing the current prototype unchanged.

6. **Migration planning**
   - Design a low-risk migration path from `/source/blurpint` into the chosen architecture.
   - Preserve current assets and working content while preventing duplicated sources of truth.

## Strategic Outputs

Create all of the following:

- `docs/architecture/01-foundation-audit.md`
- `docs/architecture/01-gap-matrix.md`
- `docs/architecture/01-architecture-decision-record.md`
- `docs/architecture/01-migration-plan.md`
- `docs/architecture/01-route-and-asset-inventory.json`
- `docs/handovers/01-to-02.md`

If the architecture decision requires scaffolding a new application root, you may create the minimal folder structure now, but do **not** begin broad page migration yet.

## Non-Negotiable Constraints

- Do not delete `/source/blurpint` during this prompt.
- Do not choose a stack because it is fashionable; choose it because it best fits the evidence in the repo.
- Surface missing business facts explicitly instead of hiding them.
- Keep the final recommendation compatible with strong accessibility, low JavaScript overhead where possible, and future content management.

## Done When

- A future agent could read the outputs and know exactly what exists, what is missing, what architecture has been chosen, and why.
- The repo has a documented baseline for all later implementation prompts.
