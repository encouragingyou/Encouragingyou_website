# Prompt 26 - Youth Club State And First-Visit Model

## User-facing state surface

The Youth Club route inherits the shared session family state model and applies it to a first-visit, low-anxiety pathway.

### Active recurring state

- Show recurrence, time range, and next occurrence through the shared temporal engine.
- Keep `Ask to join` and `Add to calendar` visible together.

### Paused or schedule-updated state

- Reuse the shared schedule notice surface.
- Keep the contact path visible so the route still has a next step even when the live schedule changes.

### Calendar available / unavailable

- When the ICS file is available, expose the calendar action normally.
- When unavailable, rely on the shared info notice and route users back to contact rather than hiding the session entirely.

### Venue disclosure state

- Rochdale remains public.
- Exact venue detail remains enquiry-led.
- This appears in hero copy, the at-a-glance section, support copy, and FAQ answers so the rule is visible without becoming the whole page.

### No-JS state

- Core route content, breadcrumb navigation, next-step actions, and FAQ disclosure remain usable without client scripting.

### Trust-without-proof state

- The route does not depend on testimonials, participant proof, or operational claims that are not yet verified.
- Safeguarding remains visible as a live route, not as an implied internal process.

## First-visit model

The route is structured to answer four questions in order:

1. Is this space meant to feel welcoming if I do not know anyone yet?
2. What might the session actually include when I arrive?
3. Is there more than one acceptable way to take part?
4. How do I ask first if I need a calmer next step?

That ordering keeps the page youth-facing, while still making room for a parent or carer to orient themselves without taking over the route.
