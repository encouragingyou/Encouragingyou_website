# Prompt 41 To 42 Handover

Prompt 41 stabilizes the motion and interaction layer.

## What is now stable

- `site/src/styles/tokens.css` now contains the expanded motion primitives for duration, easing, distance, and stagger.
- `site/src/styles/theme.css` maps those primitives into named motion roles and reduced-motion overrides.
- `site/src/styles/utilities.css` now owns the safe reveal primitive through `data-motion="reveal"` and `data-in-view`.
- `site/src/lib/client/site-behavior.js` now handles sticky-header state, mobile-nav structural state, disclosure state sync, field-state sync, and safe in-view motion setup.
- shared components now advertise motion intent through `data-motion` instead of relying on route-local CSS guesses.
- reduced motion no longer relies on a blanket `* { transition-duration: 0.01ms }` override.

## What Prompt 42 should assume

- there is no route-transition layer to optimize or remove
- reveal behavior is intentionally subtle and should be tuned for performance, not replaced with spectacle
- the mobile nav depends on `opening/open/closing/closed` state plus delayed hiding; performance work should preserve that contract
- sticky-header polish must remain height-stable to avoid layout shift
- reduced-motion behavior is part of the public contract and should not be collapsed back into a generic hard-off rule

## Watchouts

- do not reintroduce hidden-before-JS reveal logic while chasing CLS or perceived polish
- do not turn performance work into a reason to remove focus, hover, or success/error clarity
- if asset, font, or style loading changes affect the hero or card surfaces, recheck `tests/e2e/contracts/motion-system.spec.mjs`
- trust-heavy pages should stay restrained even if later prompts add stronger motion elsewhere
