# Prompt 40 To 41 Handover

Prompt 40 stabilizes the discovery and sharing layer.

## What is now stable

- `site/src/content/discovery/default.json` is the source of truth for robots policy, sitemap rules, and social-preview families.
- `site/src/lib/content/discovery.js` is the canonical route-inventory and sitemap/robots builder.
- `site/src/lib/content/structured-data.js` holds the shared JSON-LD builders.
- `site/src/lib/content/site-content.ts` now composes page schema from shared builders instead of route-local raw objects.
- `site/scripts/generate-social-preview-assets.mjs` generates the branded route-family preview images.
- `site/src/data/generated/social-preview-manifest.json` is the runtime lookup for metadata images.
- `site/scripts/generate-discovery-files.mjs` generates `site/public/robots.txt` and `site/public/sitemap.xml`.
- `site/scripts/validate-discovery-artifacts.mjs` validates the generated discovery layer before build.

## What Prompt 41 should assume

- metadata image selection should keep flowing through `site-seo.ts`; do not add route-local OG image tags
- `robots.txt` and `sitemap.xml` are generated artifacts, not files to hand-edit
- legal/trust routes remain `noindex,follow` but may still carry dedicated share images
- safeguarding now has its own honest, non-photographic share asset and should not fall back to generic imagery
- recurring session event schema is already synchronized with the session temporal engine

## Watchouts

- do not reintroduce media-based people imagery into social preview assets
- do not add `lastmod` casually unless freshness data becomes trustworthy across the route inventory
- do not block public routes in `robots.txt` when the real intent is `noindex`
- if motion work in Prompt 41 affects hero or shell composition, keep the social preview pipeline independent from page-runtime animation logic
