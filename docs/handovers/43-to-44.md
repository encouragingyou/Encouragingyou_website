# Prompt 43 To 44

Prompt 43 introduced a live first-party analytics subsystem with a narrow statistical-purpose boundary.

## What is now stable

- `site/src/lib/analytics/contract.js` is the event and property allowlist source of truth
- `site/src/lib/analytics/preferences.js` resolves runtime measurement state, objection state, and browser privacy-signal handling
- `site/src/lib/analytics/store.js` persists aggregate daily counters in `ANALYTICS_STORAGE_DIR` or `./var/analytics`
- page views and calendar downloads are recorded server-side through `site/src/middleware.ts`
- successful non-safeguarding form submissions are recorded from `site/src/lib/server/enquiry-service.js`
- client-side CTA, contact-intent, form-start, and validation-failure events go through `site/src/pages/api/analytics.ts`
- `/cookies/` now includes the live objection control through `site/src/pages/api/analytics-preference.ts` and `site/src/components/legal/AnalyticsPreferencePanel.astro`
- `site/src/content/storageAccess/default.json`, `site/src/content/privacyNotice/default.json`, and `site/src/content/consentAwareMicrocopy/default.json` are synchronized to the new launch state

## Runtime boundary Prompt 44 must respect

- do not add any analytics fields that contain names, emails, phone numbers, message text, safeguarding detail, or raw URLs
- do not add analytics to safeguarding route interactions or safeguarding form conversion
- do not let loading, error, or empty states block analytics objection controls, legal links, or form submission
- use the existing root data attributes from `BaseLayout.astro` (`data-analytics-mode`, `data-analytics-collection`, `data-analytics-state`) if Prompt 44 needs instrumentation-aware UI decisions
- keep analytics failure silent and non-blocking

## What Prompt 44 can build on

- cookies route now exposes a real measurement-off state for the current browser
- support forms already expose `data-surface-id`
- client runtime dedupes CTA and validation events, so Prompt 44 should avoid bolting on ad hoc tracking for retry and empty-state surfaces
- the analytics docs under `docs/analytics/` define which product questions are already covered and which vanity metrics should stay out of scope

## Verification baseline after Prompt 43

- `npm run test:unit --workspaces=false`
- `npm run check --workspaces=false`
- Prompt 43 also adds the dedicated browser contract `site/tests/e2e/contracts/analytics-governance.spec.mjs`
