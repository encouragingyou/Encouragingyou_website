# Prompt 14 To 15 Handoff

Prompt 14 establishes the recurring-session domain as a real platform layer.

## What is now in place

- canonical recurring session facts now live in `site/src/content/sessions/*.json`
- `site/src/lib/domain/session-schedule.js` owns recurrence math, DST-safe timezone handling, and truthful schedule-state derivation
- `site/src/lib/domain/session-calendar.js` and `site/scripts/generate-session-calendars.mjs` generate the production ICS feeds in `site/public/calendar/`
- session cards and session detail pages consume the shared temporal engine via `site-content.ts`
- session detail routes now emit server-rendered `Event` JSON-LD through `BaseLayout`
- unit tests cover schedule math, DST boundaries, structured data, and ICS generation
- Playwright coverage now checks JSON-LD presence and calendar-feed consistency

## Prompt 15 focus

Prompt 15 should build editorial governance on top of this stable runtime instead of changing the temporal architecture again.

- treat `statusLabel`, `statusMessage`, `nextLabel`, and `scheduleNotice` as the current schedule-state copy contract
- keep recurring factual data in `sessions/*.json`
- keep session-page reassurance, FAQs, and action language in structured editorial collections
- if new editorial guardrails need fallback copy for paused or enquiry-only states, add them through structured content and route-model composition rather than inline Astro prose

## Watch-outs

- do not move schedule truth back into browser scripts or route-local copy
- do not hand-edit `site/public/calendar/*.ics`
- do not emit placeholder next-date promises for sessions that are paused, cancelled, contact-only, or missing valid recurrence inputs
- if prompt 15 renames visible schedule labels, keep the JSON-LD and calendar pipelines untouched unless the canonical content facts themselves changed
